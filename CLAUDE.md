## Project context

**10xPlantsInventory** — an AI-vision plant cataloging app (greenfield, ~3-week MVP). Users photograph plants across multiple locations; an AI provider suggests species + care info; the app schedules watering/winterization reminders. See `context/foundation/prd.md` for the full spec.

The codebase is currently the **auth scaffold only** (email/password sign-in via Supabase). The plant, location, reminder, and AI-vision features described in the PRD are **not yet built** — expect to create most of the domain from scratch.

## Commands

- `npm run dev` — dev server on the Cloudflare `workerd` runtime (not plain Node)
- `npm run build` — production SSR build via `@astrojs/cloudflare`
- `npm run preview` — preview the production build
- `npm run lint` / `npm run lint:fix` — ESLint with type-checked rules
- `npm run format` — Prettier (includes `prettier-plugin-astro` + `prettier-plugin-tailwindcss`)
- `npx astro sync` — regenerate `astro:*` types; **run this after changing `astro.config.mjs` or content collections**, otherwise lint/build fails on missing generated types (CI runs it before lint)
- `npx wrangler deploy` — deploy to Cloudflare Workers

There is **no test runner configured** — no `npm test` script, no test framework installed.

Pre-commit: husky + lint-staged runs `eslint --fix` on `*.{ts,tsx,astro}` and `prettier --write` on `*.{json,css,md}`.

## Architecture

Astro 6 SSR app with React 19 islands, Tailwind 4, Supabase auth, and shadcn/ui. Deployed to Cloudflare Workers.

### Rendering

`output: "server"` (astro.config.mjs) — every route is server-rendered. `prerender` defaults to `false`; do not add `export const prerender = false` to API routes, it is redundant here.

### Auth flow

- `src/lib/supabase.ts` — `createClient(headers, cookies)` builds a `@supabase/ssr` cookie-session client. **It returns `null` when `SUPABASE_URL`/`SUPABASE_KEY` are unset** — every caller must null-check and degrade gracefully (the app is designed to run unconfigured).
- `src/middleware.ts` — runs on every request, resolves the user, attaches it to `context.locals.user` (typed in `src/env.d.ts`). Redirects unauthenticated requests for any path in the `PROTECTED_ROUTES` array to `/auth/signin`. Add new protected paths there.
- API endpoints: `src/pages/api/auth/{signin,signup,signout}.ts` — `POST` handlers that read `request.formData()` and `context.redirect(...)` with errors passed back as `?error=` query params.
- Auth pages: `src/pages/auth/{signin,signup,confirm-email}.astro`; `src/components/auth/*` holds the React form islands.

### Environment & secrets

`SUPABASE_URL` and `SUPABASE_KEY` are declared in the `astro.config.mjs` `env.schema` as `context: "server", access: "secret", optional: true`. Import them only via `astro:env/server`. Local config: `.env` for the Node toolchain, `.dev.vars` for the Cloudflare local runtime (both gitignored — copy from `.env.example`). Local Supabase: `npx supabase start` (needs Docker).

## Conventions

- **Path alias**: `@/*` maps to `./src/*`.
- **Astro vs React**: Astro components for static content/layout; React components only where interactivity is needed. No Next.js directives (`"use client"` etc.).
- **Tailwind 4** is configured CSS-first via the `@tailwindcss/vite` plugin and `src/styles/global.css` — there is no `tailwind.config` file.
- **Class merging**: use the `cn()` helper from `@/lib/utils` (clsx + tailwind-merge); never concatenate class strings by hand.
- **shadcn/ui**: "new-york" style, components in `src/components/ui/`. Add new ones with `npx shadcn@latest add [name]`. Icon library is `lucide-react`.
- **Lint is strict and type-aware** (`strictTypeChecked` + `stylisticTypeChecked`). `react-compiler/react-compiler` is set to `error` — keep components compiler-safe (no mutation of props/state, follow the rules of hooks). Prefix intentionally-unused vars with `_`.
- **Placement**: services/helpers in `src/lib/`; shared types/DTOs in `src/types.ts`; React hooks in `src/components/hooks/`.
- **Supabase migrations** (none exist yet): `supabase/migrations/`, named `YYYYMMDDHHmmss_short_description.sql`. Per the PRD's per-user isolation guardrail, enable RLS on every new table with granular per-operation, per-role policies.

## 10xDevs AI Toolkit - Module 2, Lesson 1

Move from sprint-zero setup to project orchestration with the **roadmap chain**:

```
(Module 1 foundation docs) -> /10x-roadmap -> backlog-ready roadmap items
```

`/10x-roadmap` is the lesson focus. `/10x-new` is intentionally introduced in Module 2, Lesson 2, when a selected roadmap item becomes an implementation change folder.

### Task Router - Where to start

| Skill | Use it when |
| --- | --- |
| **Roadmap (lesson focus)** | |
| `/10x-roadmap` | You have `context/foundation/prd.md` and a scaffolded project baseline, and you need a vertical-first MVP roadmap. The skill reads the PRD, inspects the code baseline, uses available foundation docs such as `tech-stack.md`, `infrastructure.md`, and `deploy-plan.md`, then writes `context/foundation/roadmap.md`. Use it BEFORE creating per-change folders or implementation plans. |
| **Re-run upstream if needed** | |
| `/10x-shape` / `/10x-prd` / `/10x-tech-stack-selector` / `/10x-bootstrapper` / `/10x-agents-md` / `/10x-infra-research` | Bundled from Module 1 so foundation contracts can be fixed before roadmap sequencing. If roadmap generation exposes a PRD gap, repair the PRD before pretending the backlog is ready. |

### How the chain hands off

- `/10x-roadmap` bridges product and implementation. It does not choose frameworks, design schemas, or write a per-change implementation plan.
- The output is `context/foundation/roadmap.md`: ordered milestones, vertical slices, bounded foundations, dependencies, unknowns, risk, and backlog handoff fields.
- Roadmap items should receive stable human-readable identifiers in backlog tools. The actual `context/changes/<change-id>/` folder is created in Lesson 2 with `/10x-new`.

### Roadmap boundaries

- Default to vertical slices: user-visible outcomes that cross UI, data, business logic, and integrations.
- Horizontal work is allowed only as a bounded enabler that names the downstream vertical milestone it unlocks.
- Avoid orphan horizontal work such as "build the whole database", "build all API endpoints", or "design the whole UI" before the first user-visible flow.
- Roadmap is not a calendar estimate. Do not invent dates, story points, or sprint velocity unless the user explicitly asks for a separate planning artifact.

### Foundation paths used by this lesson

- `context/foundation/prd.md` - input
- `context/foundation/tech-stack.md` - optional input
- `context/foundation/infrastructure.md` - optional input
- `context/deployment/deploy-plan.md` - optional input
- `context/foundation/roadmap.md` - output
- `context/foundation/lessons.md` - recurring rules and pitfalls
- `docs/reference/contract-surfaces.md` - load-bearing names registry

Skills must not write to `context/archive/`. Archived changes are immutable; if a resolved target path starts with `context/archive/`, abort with: "This change is archived. Open a new change with `/10x-new` instead."
