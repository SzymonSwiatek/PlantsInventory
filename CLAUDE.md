## Project context

**10xPlantsInventory** â€” an AI-vision plant cataloging app (greenfield, ~3-week MVP). Users photograph plants across multiple locations; an AI provider suggests species + care info; the app schedules watering/winterization reminders. See `context/foundation/prd.md` for the full spec.

The codebase is currently the **auth scaffold only** (email/password sign-in via Supabase). The plant, location, reminder, and AI-vision features described in the PRD are **not yet built** â€” expect to create most of the domain from scratch.

## Architecture

Astro 6 SSR app with React 19 islands, Tailwind 4, Supabase auth, and shadcn/ui. Deployed to Cloudflare Workers.

### Rendering

`output: "server"` (astro.config.mjs) â€” every route is server-rendered. `prerender` defaults to `false`; do not add `export const prerender = false` to API routes, it is redundant here.

### Auth flow

- `src/lib/supabase.ts` â€” `createClient(headers, cookies)` builds a `@supabase/ssr` cookie-session client. **It returns `null` when `SUPABASE_URL`/`SUPABASE_KEY` are unset** â€” every caller must null-check and degrade gracefully (the app is designed to run unconfigured).
- `src/middleware.ts` â€” runs on every request, resolves the user, attaches it to `context.locals.user` (typed in `src/env.d.ts`). Redirects unauthenticated requests for any path in the `PROTECTED_ROUTES` array to `/auth/signin`. Add new protected paths there.
- API endpoints: `src/pages/api/auth/{signin,signup,signout}.ts` â€” `POST` handlers that read `request.formData()` and `context.redirect(...)` with errors passed back as `?error=` query params.
- Auth pages: `src/pages/auth/{signin,signup,confirm-email}.astro`; `src/components/auth/*` holds the React form islands.

### Environment & secrets

`SUPABASE_URL` and `SUPABASE_KEY` are declared in the `astro.config.mjs` `env.schema` as `context: "server", access: "secret", optional: true`. Import them only via `astro:env/server`. Local config: `.env` for the Node toolchain, `.dev.vars` for the Cloudflare local runtime (both gitignored â€” copy from `.env.example`). Local Supabase: `npx supabase start` (needs Docker).

## Conventions

- **Path alias**: `@/*` maps to `./src/*`.
- **Astro vs React**: Astro components for static content/layout; React components only where interactivity is needed. No Next.js directives (`"use client"` etc.).
- **Tailwind 4** is configured CSS-first via the `@tailwindcss/vite` plugin and `src/styles/global.css` â€” there is no `tailwind.config` file.
- **Class merging**: use the `cn()` helper from `@/lib/utils` (clsx + tailwind-merge); never concatenate class strings by hand.
- **shadcn/ui**: "new-york" style, components in `src/components/ui/`. Add new ones with `npx shadcn@latest add [name]`. Icon library is `lucide-react`.
- **Lint is strict and type-aware** (`strictTypeChecked` + `stylisticTypeChecked`). `react-compiler/react-compiler` is set to `error` â€” keep components compiler-safe (no mutation of props/state, follow the rules of hooks). Prefix intentionally-unused vars with `_`.
- **Placement**: services/helpers in `src/lib/`; shared types/DTOs in `src/types.ts`; React hooks in `src/components/hooks/`.
- **Supabase migrations** (none exist yet): `supabase/migrations/`, named `YYYYMMDDHHmmss_short_description.sql`. Per the PRD's per-user isolation guardrail, enable RLS on every new table with granular per-operation, per-role policies.

## 10xDevs AI Toolkit - Module 2, Lesson 2

Turn one roadmap item into the first implementation cycle with the **change planning chain**:

```
/10x-roadmap -> /10x-new -> /10x-plan -> /10x-plan-review -> /10x-implement
```

`/10x-new`, `/10x-plan`, `/10x-plan-review`, and `/10x-implement` are the lesson focus. `/10x-frame` and `/10x-research` are not required rituals here; they are escalation paths introduced in the next lesson.

### Task Router - Where to start

| Skill | Use it when |
| --- | --- |
| **Change setup (lesson focus)** | |
| `/10x-new <change-id>` | You selected a roadmap item and need a stable change folder. Creates `context/changes/<change-id>/change.md` so planning, implementation, progress, commits, and later review all share one identity. Use AFTER roadmap selection, BEFORE `/10x-plan`. |
| **Planning (lesson focus)** | |
| `/10x-plan <change-id>` | You have a change folder and need a reviewable implementation plan. Reads roadmap context, foundation docs, codebase evidence, and any existing change notes; writes `plan.md` and `plan-brief.md` with phases, file contracts, success criteria, and `## Progress`. |
| **Plan readiness (lesson focus)** | |
| `/10x-plan-review <change-id>` | You have `plan.md` and need a light pre-code readiness check. Use it to catch missing end state, weak contracts, malformed progress, scope drift, or blind spots before code changes begin. |
| **Implementation (lesson focus)** | |
| `/10x-implement <change-id> phase <n>` | You have an approved plan and want to execute one phase with verification, manual gate, commit ritual, and SHA write-back to `## Progress`. |
| **Lifecycle closure** | |
| `/10x-archive <change-id>` | A change is merged or intentionally closed. Move it out of active `context/changes/` into archive state. |

### How the chain hands off

- `/10x-new` creates the durable change identity.
- `/10x-plan` turns that identity into an implementation contract.
- `/10x-plan-review` checks the plan before the agent mutates code.
- `/10x-implement` executes one planned phase, verifies, asks for manual confirmation when needed, commits, and records progress.

### Lesson boundaries

- Plan is the default router after roadmap selection. Start with `/10x-plan` unless the problem is unclear or external evidence is blocking.
- Do not run `/10x-frame + /10x-research` as ceremony for every change.
- Do not turn this lesson into a full end-to-end product build. A checkpoint with a planned and partially or fully implemented stream is valid.
- Code review of the implemented diff belongs to Lesson 3 via `/10x-impl-review`.
- Lifecycle closure via `/10x-archive` after a change is merged or intentionally closed.

### Paths used by this lesson

- `context/foundation/roadmap.md` - upstream roadmap
- `context/changes/<change-id>/change.md` - change identity
- `context/changes/<change-id>/plan.md` - implementation contract
- `context/changes/<change-id>/plan-brief.md` - compressed handoff
- `context/foundation/lessons.md` - recurring rules and pitfalls
- `docs/reference/contract-surfaces.md` - load-bearing names registry

Skills must not write to `context/archive/`. Archived changes are immutable; if a resolved target path starts with `context/archive/`, abort with: "This change is archived. Open a new change with `/10x-new` instead."
